Gerado em 19 Jul 2026 · Commit 788755c · Branch master

◈ Visão Geral da Arquitetura

O AI Operating System (AIOS) é uma arquitetura multi-agente distribuída, construída sobre um modelo event-driven. 8 agentes especializados comunicam-se assincronamente através de um message bus, coordenados por um orquestrador central.

Princípios Fundamentais

1
Autonomia dos Agentes
Cada agente opera independentemente e toma decisões locais dentro de seu domínio de expertise.
2
Acoplamento Frouxo
Agentes comunicam-se apenas através do message bus; sem chamadas diretas agente-a-agente.
3
Orientado a Mensagens
Todas as interações são mensagens assíncronas, permitindo execução paralela non-blocking.
4
Isolamento de Falhas
Falhas de agentes são contidas e repostas sem derrubar o sistema.
5
Observabilidade
Cada mensagem, decisão e mudança de estado é registrada e rastreável.

Algoritmo de Roteamento

function route(task):
  candidates = registry.find_by_capability(task.required_capabilities)
  candidates = filter(candidates, status == READY)
  candidates = sort(candidates, by=score(task, agent))
  if candidates is empty:
    queue(task, retry_after=30s)
    return
  best = candidates[0]
  assign(task, best)

⬡ Orquestrador

O cérebro do sistema. Analisa pedidos, decompõe em subtarefas, roteia para agentes e agrega resultados.

1
Análise de Tarefa
Parse da requisição para entender intenção e requisitos. Classificação de prioridade (critical, high, medium, low).
2
Decomposição
Quebra de tarefas complexas em subtarefas com dependências. Máximo 4 agentes paralelos.
3
Roteamento
Match de subtarefas com agentes baseado em capabilities e disponibilidade. Strategy: capability_based.
4
Agendamento
Ordem de execução respeitando dependências. Execução paralela quando possível.
5
Agregação
Coleta de resultados dos agentes e composição da saída final.
6
Tratamento de Erros
Detecção de falhas, retry, re-roting ou escalonamento. Circuit breaker para agentes com alta taxa de falha.

Message Bus Patterns

Publish/Subscribe
Broadcast de mensagens para todos os subscribers. Usado para health broadcasts e notificações de estado.
Request/Reply
Chamadas síncronas com correlation IDs. Usado para queries entre agentes e respostas diretas.
Task Queue
Filas duráveis para distribuição de trabalho. Redis Streams com at-least-once delivery e dead letter queue.

Fault Tolerance

CenárioEstratégiaDetalhe
Agent crashHeartbeat detectionTarefa re-rota para outro agente via heartbeat perdido
Message lossRedis StreamsAt-least-once delivery; idempotency keys previnem duplicação
TimeoutConfigurable timeoutTarefas com timeout configurável; excedidas são canceladas e repostas
High failure rateCircuit breakerAgentes com alta taxa de falha temporariamente offline

Passos do Workflow

1. Recepção da Requisição

Usuário submete tarefa via REST API. API Gateway valida autenticação, aplica rate limiting e valida input.

2. Análise e Decomposição

Orchestrator analisa a requisição, entende intenção e requisitos, e decompõe em subtarefas menores com dependências.

3. Roteamento

Task Router consulta o Agent Registry para encontrar agentes com capacidades adequadas. Aplica scoring baseado em capability match, load e success rate.

4. Execução Paralela

Subtarefas são distribuídas para agentes via Message Bus. Agentes executam independentemente de forma assíncrona e paralela.

5. Coleta de Resultados

Orchestrator coleta resultados via Message Bus. Monitora timeouts e falhas. Aplica retry/re-route quando necessário.

6. Agregação e Entrega

Resultados são mergeados, conflitos resolvidos, e resposta formatada é devolvida ao usuário via API Gateway.
